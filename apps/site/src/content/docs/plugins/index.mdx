Buntime ships a set of core plugins under `plugins/`. They are ordinary plugins
(nothing special compared to ones you write) and demonstrate every part of the
[plugin system](/concepts/plugin-system/): persistent routes, service sharing,
request hooks, and micro-frontend UIs.

## Bundled plugins

| Plugin | Base path | Default | Purpose |
|--------|-----------|---------|---------|
| [`plugin-gateway`](/plugins/gateway/) | `/gateway` | **enabled** | CORS, rate limiting, the micro-frontend app-shell, and SSE monitoring |
| [`plugin-proxy`](/plugins/proxy/) | `/redirects` | **enabled** | Dynamic reverse proxy and redirect rules |
| [`plugin-turso`](/plugins/turso/) | _(service)_ | **enabled** | Durable SQL via Turso; exposed to other plugins through `provides()` |
| [`plugin-keyval`](/plugins/keyval/) | `/keyval` | opt-in | Deno-KV-like key/value store, queues, and full-text search |
| [`plugin-logs`](/plugins/logs/) | `/logs` | opt-in | In-memory logs with SSE streaming |
| [`plugin-metrics`](/plugins/metrics/) | `/metrics` | opt-in | Prometheus metrics + SSE for dashboards |
| [`plugin-vhosts`](/plugins/vhosts/) | _(hostname)_ | opt-in | Maps hostnames/subdomains to apps |

<Aside type="note">
  `plugin-turso` has no `base` — it's a service provider. Other plugins reach
  its `TursoService` via `ctx.getPlugin("@buntime/plugin-turso")`.
  `plugin-vhosts` is a hook-only plugin that routes by `Host` header rather than
  a path prefix.
</Aside>

## Enabling a plugin

Default state is set by `enabled` in each plugin's `manifest.yaml`. Opt-in
plugins can be turned on at runtime without a restart:

```bash
curl -X POST http://localhost:8000/api/plugins/%40buntime%2Fplugin-keyval/enable
```

The change edits the manifest on disk, so it survives restarts. See
[Plugin System → Hot reload](/concepts/plugin-system/#hot-reload).
